STUDY EDITION · FOR YOU, NOT THE JURY
Start Here
Your project is three programs

In one sentence: Fakerni lets a family share live shopping/task lists (“Fakras”) across web and mobile, with AI, budgets and notifications. The same backend serves both apps.

1 · Backend

The “brain”. Stores data, enforces rules, exposes an API. Django + DRF + Channels. Folders: users, household, fakras.

2 · Web app

Runs in a browser. React + Vite. Folder: web/. Talks to the backend over the internet.

3 · Mobile app

Runs on a phone. React Native + Expo. Folder: mobile/. Same logic as web, native features added.

The web and mobile apps are just faces. They hold no data — they ask the backend for everything.

Vocabulary
Words the jury will use
API
A menu of requests the backend accepts. The apps “order” from it (get fakras, add item…).
REST endpoint
One URL for one action, e.g. POST /api/fakras/ = create a fakra.
JWT (token)
A signed digital badge proving who you are. Sent with every request instead of a password.
WebSocket
A phone line kept open so the server can push updates instantly (real-time).
ORM / Model
Python classes that map to database tables. You write Python, Django writes SQL.
Serializer
Translator between database objects and the JSON the apps send/receive.
Migration
A versioned change to the database shape (add a column, a table…).
Consumer
The WebSocket equivalent of a view — handles a live connection.

The Domain
The idea, as a real family

User = a person with an account.

Household = the family group. You join with an invite code.

Membership = your place in a household + your role (owner / admin / member).

Fakra = one list (e.g. “Weekly Groceries”). Can belong to the family or be personal.

Item = one line in the list (milk, ×2, done by Ali).

Analogy: a Household is a shared fridge door. Each Fakra is a sticky note on it. Each Item is a line on the note. Anyone in the family can add a line or tick one off — and everyone sees it change at the same moment.

FAKERNI · STUDY05 / 17
Walkthrough
What happens when you add an item
1

You tap “Add” in the app

The app sends POST /api/fakras/8/items/ with the item data + your JWT badge.

2

Django checks your badge & access

require_fakra_access() confirms you’re allowed to touch fakra 8. If not → refused.

3

The item is saved

The serializer validates the data and the ORM writes a new row to the Item table.

4

Everyone is told — live

The view calls broadcast_to_fakra() (WebSocket) + a push notification to other members.

5

The screen updates

Your app shows the new item; other people’s apps receive the live event and refresh instantly.

FAKERNI · STUDY06 / 17
Walkthrough
How login & tokens work
1

You send email + password

POST /api/token/. The backend checks the password (compared against a PBKDF2 hash, never plaintext).

2

You get two tokens

An access token (valid 60 min) and a refresh token (valid 7 days).

3

Every request carries the access token

Sent as Authorization: Bearer …. The server trusts the signature — no DB lookup needed.

4

When it expires, it refreshes silently

The app catches the “401 expired”, uses the refresh token to get a new access token, and retries — you never notice.

FAKERNI · STUDY07 / 17
Walkthrough
How real-time actually works
1

The app opens a WebSocket

e.g. ws/fakras/8/?token=… — a line that stays open, with the JWT in the URL.

2

Middleware authenticates it

JWTAuthMiddleware validates the token and attaches the user; the consumer then checks access and joins a “group”.

3

Something changes

When any member adds/completes an item, the view calls broadcast_to_fakra().

4

The message fans out

Redis pushes the event to everyone in that group; each open app receives it and updates — no refresh.

FAKERNI · STUDY08 / 17
Walkthrough
How “Smart Add” uses AI
1

You type free text

“milk, eggs and 2kg rice”. The app sends it to POST /…/items/smart-add/.

2

The backend calls Gemini

ai.py sends your text + a system prompt + a strict schema describing the item shape.

3

Gemini returns clean JSON

Because of the schema, the answer is always valid structured data — never messy text to guess-parse.

4

Items are created (and editable)

Each parsed item is saved to the fakra. If the AI key is missing/fails → a clean error, app keeps working.

FAKERNI · STUDY09 / 17
Web App
How web/ is organised

src/api/

One file per resource (fakras.js, households.js…). Thin functions that call endpoints. client.js attaches the JWT & auto-refreshes it.

src/pages/

One file per screen: Dashboard, FakraDetail, Analytics, login, profile… Each fetches data via the api layer and renders it.

src/context/

AuthContext holds “who is logged in”, exposes login/logout, and guards protected pages.

src/hooks/

useFakraSocket / useUserSocket open WebSockets and refresh the screen on live events.

Mental model: pages ask the api layer → api layer hits the backend → results render. Live updates arrive through hooks.

FAKERNI · STUDY10 / 17
Mobile App
How mobile/ mirrors web

app/

Screens as files (expo-router). (auth) group = login flow; (tabs) = Fakras / Households / Analytics / Profile; fakras/[id] = detail.

src/api/

Same per-resource files as web. client.js stores tokens in AsyncStorage instead of the browser.

Shared logic

Auth, hooks, api and i18n are near-identical to web — that’s why building two apps wasn’t double the work.

Native extras

Things a browser can’t do: camera barcode scan, biometric lock, native push tokens.

One-liner for the jury: “The mobile app reuses the web app’s architecture and API layer; only the presentation and device features differ.”

Security — in plain words
Why each measure exists

Hashed passwords — even if the DB leaks, real passwords stay unreadable.

JWT expiry — a stolen token is useless after 60 minutes.

Rate limiting — blocks robots guessing passwords (too many tries → 429).

OTP reset — one-time code, expires in 10 min, doesn’t reveal if an email exists.

Encryption at rest — OTP codes & push tokens are stored scrambled.

Access checks everywhere — you can never open a family’s data that isn’t yours.

The golden rule of the app: the backend never trusts the client. Every request re-proves who you are and what you may touch.

FAKERNI · STUDY15 / 17
Decode the Hard Questions
If the jury asks…

“Where is the business logic?” → in the backend views.py of each app — never in the phone/browser.

“How do two users not see each other’s data?” → every request runs require_fakra_access() / role checks.

“What makes it real-time?” → open WebSockets + Redis fan-out; the server pushes, the app doesn’t poll.

“Is the AI reliable?” → structured-output schema forces valid JSON; results are shown as editable, and it degrades gracefully.

“Will it scale?” → stateless JWT API (add servers) + Redis channel layer (real-time across them).

“Hardest part?” → authenticated, spoof-proof real-time across three channel scopes.
